# Taiko Docs Revamp — Design Spec

## Goal

Replace the current Taiko docs site (`packages/docs-site`, Astro/Starlight) with a new docs site built on Vocs, designed agent-first. Inspired by Tempo's docs pattern — agents are the primary audience, humans second.

## Audience Priority

1. **AI coding agents** — need RPC endpoints, chain IDs, contract addresses, EVM differences, and actionable commands
2. **App developers** — deploying contracts, bridging tokens, adding Taiko to dApps
3. **Protocol researchers** — understanding based rollup design, proving system, economics

## Platform

**From**: Astro + Starlight (current)
**To**: Vocs (React + Vite docs framework by wevm)

### Why Vocs

- Built-in `llms.txt` / `llms-full.txt` auto-generation
- Every page accessible as raw markdown via `.md` URL extension
- Content visibility directives (`:::show[html]` / `:::hide[md]`) for agent vs human content
- AI CTA dropdown ("Open in ChatGPT")
- Native to the Ethereum/web3 ecosystem (powers Wagmi, Viem docs)
- Excellent code blocks: focus, highlight, diff, code-groups, twoslash
- Steps, callouts, cards built in
- Dark mode by default
- Deploys to Vercel as static site

### Trade-offs Accepted

- Smaller community than Starlight (~1.4k stars)
- No built-in i18n (not currently needed)
- Ships React runtime (acceptable for docs site)
- No built-in versioning (not currently needed)

## Agent Layer — Three Tiers

### Tier 1: SKILL.md

Static file at `docs.taiko.xyz/SKILL.md` containing everything an agent needs in one fetch:

- YAML frontmatter with name, description, and trigger keywords
- Quick reference table (chain IDs, RPCs, explorers for mainnet + testnet)
- How Taiko differs from Ethereum (Shanghai EVM, no sequencer)
- EVM Compatibility section with Foundry profile configuration
- Tooling overview (Foundry, Hardhat, viem, ethers.js, cast)
- Contract addresses — only developer-relevant ones (Bridge, ERC20Vault, SignalService, TaikoToken, WETH, USDC)
- Common task commands (deploy, verify, read state, add network, bridge)
- Behavioral rules for agents
- Links to full documentation

**Eval results**: Tested against 5 scenarios. With skill: 20/20 assertions pass (100%). Without skill: 10/20 (50%). Critical wins on Shanghai EVM version, current testnet (Hoodi, not deprecated Hekla), and contract addresses.

### Tier 2: llms.txt / llms-full.txt

Auto-generated by Vocs — zero configuration:

- `llms.txt`: index of all pages with one-sentence descriptions
- `llms-full.txt`: entire docs site as plain text
- Every page at `/<path>.md` for raw markdown access

### Tier 3: Dedicated Agent Quickstart Page

At `/quickstart/agent`, with code-groups for different agent platforms (Claude Code, Cursor, Codex CLI, generic). Includes starter prompts and a machine-readable docs reference table.

## Agent/Human Tab Pattern

Task-oriented guide pages use Vocs `:::code-group` to present an **Agent tab** (shown first, default) with a one-liner prompt, alongside a **Human tab** that directs to the manual steps below. This follows Tempo's "Prompt mode / Manual mode" pattern.

### Where This Pattern Applies

Only on pages where there is a clear, self-contained task an agent can accomplish with the SKILL.md context:

| Page                              | Agent Prompt                                                                                           |
| --------------------------------- | ------------------------------------------------------------------------------------------------------ |
| `/quickstart/deploy`              | `Read https://docs.taiko.xyz/SKILL.md and deploy my contract at src/MyContract.sol to Taiko`           |
| `/quickstart/connect`             | `Read https://docs.taiko.xyz/SKILL.md and add Taiko mainnet and testnet to my wagmi config`            |
| `/guides/deploy-a-contract`       | `Read https://docs.taiko.xyz/SKILL.md and deploy and verify src/MyContract.sol on Taiko mainnet`       |
| `/guides/bridge-tokens`           | `Read https://docs.taiko.xyz/SKILL.md and bridge ERC20 tokens from Ethereum to Taiko programmatically` |
| `/guides/verify-a-contract`       | `Read https://docs.taiko.xyz/SKILL.md and verify my contract at <ADDRESS> on Taiko`                    |
| `/guides/interact-with-contracts` | `Read https://docs.taiko.xyz/SKILL.md and read the balance of <ADDRESS> on Taiko`                      |

### Where This Pattern Does NOT Apply

- `/network/*` — reference data, not tasks
- `/protocol/*` — conceptual content, not actionable tasks
- `/resources/*` — reference material
- `/guides/run-a-node` — too complex/infrastructure-heavy for a one-liner prompt
- `/guides/enable-a-prover` — requires hardware setup, not a prompt task

### Implementation in Vocs

Each applicable page opens with:

    :::code-group

    ```txt [Agent]
    Paste this into your coding agent:

    Read https://docs.taiko.xyz/SKILL.md and deploy my contract at src/MyContract.sol to Taiko mainnet
    ```

    ```txt [Human]
    Follow the step-by-step guide below.
    ```

    :::

Then the full manual steps follow using `::::steps`, `:::code-group` (for mainnet/testnet variants), `:::warning` callouts, etc.

## Page Writing Guidelines

### Voice & Tone

- Imperative, concise, direct — "Add a Taiko profile to your foundry.toml" not "You can add..."
- Agent-first means every sentence either teaches or provides a value — no filler
- Tables for reference data, steps for procedures, code blocks with highlighted lines for commands

### Vocs Features Used

| Feature                             | Purpose                                             |
| ----------------------------------- | --------------------------------------------------- |
| `::::steps`                         | Numbered procedural guides (auto-numbered)          |
| `:::code-group`                     | Multi-tool/multi-network tabbed code blocks         |
| `:::warning` / `:::info` / `:::tip` | Semantic callouts for critical info                 |
| `// [!code hl]`                     | Highlight important lines in code blocks            |
| `// [!code focus]`                  | Dim surrounding code to focus on key lines          |
| Code block titles `[foundry.toml]`  | File path headers on code blocks                    |
| `:::show[html]` / `:::hide[md]`     | Content visibility for agents vs humans (sparingly) |
| `<Cards>` / `<Card>`                | Navigation cards on landing/overview pages          |

### Content Visibility Directives

Used sparingly — only where agent and human content genuinely diverges:

- On `/quickstart/agent`: `:::hide[md]` wraps a message telling LLMs to fetch SKILL.md
- On guide pages: NOT used — the imperative prose and code blocks work well for both audiences

## Content Sources

All content is rewritten from scratch for accuracy and agent-friendliness. Technical details verified against:

- Current docs (packages/docs-site/src/content/docs/) — for structure and coverage
- Protocol deployment logs (packages/protocol/deployments/) — for contract addresses
- Network configuration files — for RPC endpoints, chain IDs
- Protocol source code — for bridging flow, contract interfaces

## Success Criteria

1. **Agent accuracy**: An agent with SKILL.md correctly deploys, bridges, and configures Taiko (validated by eval — 100% assertion pass rate)
2. **Agent discoverability**: llms.txt, llms-full.txt, .md URLs all auto-generated by Vocs
3. **Human usability**: Clean dark-mode site with step-by-step guides, code-groups, and callouts
4. **Content accuracy**: All chain IDs, RPC URLs, contract addresses verified against actual deployments
5. **Production ready**: Deployable to Vercel, domain-ready, all pages complete
