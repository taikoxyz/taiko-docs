# Proving System [Taiko's multi-proof verification system.]

Taiko uses a **multi-proof architecture** to verify the correctness of L2 state transitions. Rather than relying on a single proof type, the protocol requires multiple independent proof systems to agree on each batch of blocks. This design provides defense in depth -- even if one proof system is compromised, the other prevents invalid state transitions from being accepted.

## Multi-Proof Architecture

The core principle is simple: **no single proof type is trusted in isolation**. Every batch must be verified by at least two independent proof systems before it can be considered proved.

The current configuration requires:

| Proof Type | Implementation | Trust Basis |
|-----------|----------------|-------------|
| **sgxReth** (Raiko) | SGX TEE running a Rust Ethereum execution client | Hardware-based trusted execution environment |
| **sgxGeth** (Gaiko) | SGX TEE running a Go Ethereum execution client | Hardware-based trusted execution environment (different client) |

This provides both **proof diversity** (two independent implementations) and **client diversity** (Rust vs. Go execution clients), significantly reducing the risk of correlated failures.

:::info
The protocol is designed to support ZK proofs (RISC0, SP1) as additional or alternative proof types. When ZK proofs are active, the configuration can be TEE + ZK rather than TEE + TEE, providing an even stronger security guarantee since the two proof types rely on fundamentally different trust assumptions.
:::

## ComposeVerifier: Orchestrating Multiple Proofs

The **ComposeVerifier** contract is the central coordination point for multi-proof verification. It designates which sub-verifiers are active in the protocol and ensures that a sufficient set of sub-proofs are verified before the overall proof is accepted.

The verification flow:

1. A prover generates proofs from each required proof system for a batch (or set of batches).
2. The prover calls `proveBatches` on TaikoInbox, passing the aggregated proof.
3. TaikoInbox forwards the proof to ComposeVerifier.
4. ComposeVerifier decodes the aggregated proof into individual sub-proofs and routes each one to its respective sub-verifier (e.g., SgxVerifier for SGX proofs).
5. ComposeVerifier checks that the set of verified sub-proofs is **sufficient** according to the protocol's current configuration.
6. If all required sub-proofs are valid, the batch is marked as proved.

The `areVerifiersSufficient` method on ComposeVerifier determines whether a given set of verifiers meets the protocol's requirements. This allows the protocol to upgrade its proof requirements without changing the core verification logic.

## SGX Proofs

SGX (Software Guard Extensions) proofs use Intel's Trusted Execution Environment (TEE) technology. An SGX enclave re-executes the L2 block transitions in a hardware-isolated environment and produces a signed attestation that the state transition is correct.

### How SGX Verification Works

1. An SGX enclave loads the L2 block data and re-executes all transactions.
2. The enclave produces a signature over the resulting state transition using an ECDSA key pair generated inside the enclave.
3. The signature and instance ID are submitted on-chain.
4. The **SgxVerifier** contract validates the signature against a registered SGX instance.

### SGX Instance Management

SGX instances are registered on-chain via the SgxVerifier contract. Key parameters:

| Parameter | Value | Description |
|-----------|-------|-------------|
| **Instance expiry** | 365 days | SGX instances must be re-registered annually. |
| **Validity delay** | 1 hour | New instances cannot be used immediately after registration. |

Instances can be added by the contract owner and removed by the owner or a designated watchdog role. If an invalid proof is detected, the SGX instance is automatically rotated.

### Batch Proof Verification

SgxVerifier supports batch proof verification via `verifyBatchProof`, which validates a single signature covering multiple block state transitions. This is more gas-efficient than verifying each block individually.

## ZK Proofs

The protocol includes verifier contracts for two ZK proof systems:

- **RISC0 Verifier** -- verifies proofs generated by the RISC Zero zkVM
- **SP1 Verifier** -- verifies proofs generated by the Succinct SP1 prover

ZK proofs provide a **cryptographic guarantee** of correctness that does not depend on hardware trust assumptions. They are computationally more expensive to generate than SGX proofs but provide a stronger security model.

When active, a typical configuration would pair an SGX proof (fast, lower cost) with a ZK proof (slower, higher cost, stronger guarantee). The ComposeVerifier treats both as equally valid sub-proofs.

## End-to-End Verification Flow

```
Proposer submits batch
        |
        v
  Batch is PROPOSED (on-chain, unproven)
        |
        v
  Prover generates sgxReth proof  +  Prover generates sgxGeth proof
        |                                      |
        v                                      v
  Sub-proofs aggregated into single proof submission
        |
        v
  proveBatches() called on TaikoInbox
        |
        v
  ComposeVerifier routes sub-proofs to respective verifiers
        |
        v
  All required sub-proofs verified --> Batch is PROVED
        |
        v
  verifyBatches() checks sequential chain from genesis
        |
        v
  Batch is VERIFIED (final)
```

## Proving Windows

Under the Pacaya fork, the proving and cooldown windows are configured as follows:

| Window | Duration | Purpose |
|--------|----------|---------|
| **Proving window** | 2 hours | Time allowed for provers to submit proofs after a batch is proposed. |
| **Cooldown window** | 2 hours | Waiting period after a proof is submitted before the batch can be verified. |

:::warning
If no proof is submitted within the proving window, the proposer's liveness bond may be forfeited. Provers have a strong economic incentive to submit proofs promptly.
:::

## Liveness Bonds

Proposers must deposit a liveness bond when proposing a batch. The base bond is **125 TAIKO tokens** per batch. This bond serves two purposes:

1. **Spam prevention** -- makes it economically costly to propose invalid or empty batches at scale.
2. **Prover incentive alignment** -- the bond is returned to the proposer once the batch is proved and verified. If the batch is never proved, the bond can be forfeited.

## Current Proof Coverage

Taiko currently runs with **100% proof coverage** on mainnet. In other words, every proposed batch is expected to be covered by the active proof configuration before it can progress to final verification.

You can monitor the current proving status and coverage on [proofs.taiko.xyz](https://proofs.taiko.xyz/).

## Summary

Taiko's proving system achieves security through redundancy:

- **Multiple proof types** must independently verify each batch
- **Client diversity** (different execution clients) reduces correlated failure risk
- **ComposeVerifier** provides a flexible framework for adding or replacing proof systems
- **SGX proofs** offer fast, hardware-based verification
- **ZK proofs** provide the strongest cryptographic guarantees
- **Short proving windows** (2 hours) keep the protocol responsive while giving provers sufficient time
